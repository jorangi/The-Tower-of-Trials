### const의 어려움
const는 그냥 상수정도로만 C#에서는 사용했던 것 같다.<br>
C++에서는 물론 안써도 프로그램 자체에 지장은 없겠지만, Effective C++도 읽었고 나름 const를 쓰는 것을 권장한다고 하니<br>
써보고자 노력하는데, const 관련 컴파일러 지침이 굉장히 까다롭고 복잡하다는 것을 알 수 있었다.<br>
가령 ToString() 함수는 내부에서 변수를 변경하거나 하지 않으니, const 함수로 만들었다. <br>
그런데, ToString에서 GetValue()라는 함수를 호출해야 하는데, GetValue는 내부 변수를 변경하는 로직이 있어 const 함수가 아니다<br>
이럴 경우 const 함수지만 특정 변수는 변경할 수 있다는 예외처리를 해주는 mutable을 변수에 작성해준다.<br>
이 경우 const 함수에서도 변수를 변경할 수 있게된다.<br>
따라서 ToString()에서 이제 const 함수인 GetValue()를 호출할 수 있게된다.<br>
사실 const는 컴파일러단에서 변수의 값 변조를 하지 말라는 일종의 약속이고, 물론 외부 변조도 방지할 수는 있겠지만<br>
가장 큰 사용 목적은 프로그래머에게, 이 변수는 변경하지 마시오라는 의미이다.<br>
지금이야 1인 프로젝트지만 다인 프로젝트에서 누가 잘못 건드리면 터질 수 있으니 방지해주는 안전장치인 셈이다.

- ### Header-only
    - 사실 c#하다가 cpp하니까 너무 적응이 어렵다.
    - c#은 그냥 클래스 만들고 객체 만들면 땡이다.<br>물론 유니티에서 Assembly-Csharp로 해주니까 그런거긴 한데...
    - cpp에서는 헤더 파일로 클래스 만들고, cpp 파일로 구현한다.
        - Header-only는 cpp 굳이 안써도 되면 쓰지 말자는 취지인 것 같다.
        - 장점이 몇 개 있긴 한데, 굳이 cpp파일로 구현부 없애니 가독성은 좋다. 선언이랑 간단한 구현부뿐이니까.
        - 굳이 따지면 abstract class 같은 느낌이 비슷하려나? interface에 가깝다는 느낌도 들긴 한데, interface는 구현은 아예 불가능하니까
        - cpp는 결국 다른 라이브러리?를 가져다가 쓰려면 헤더를 include해야 한다. 결국 헤더는 필수적이고
        - .cpp파일에 복잡한 구현을 하는데, 이 프로젝트에서는 ModifiableStat.cpp나 StatModifier.cpp가 그렇다.
        - 그런데, 막 엄청 복잡한 기능이 아닌 경우에는 Header-only에 구현하는 걸 추천하는 경우가 많은 것 같다.
        - 그 예시가 Components.h이다. 내부에 Component랑 HealthComponent 정도만 넣어놓긴 했는데, 별도의 구현용 .cpp 파일은 없다.
    - 찾아보니 .hpp라는 cpp전용 헤더 확장자도 있는 것 같긴 한데, 그냥 .h쓰는듯. 이유는 큰 차이가 없고 기존 .h 기반의 인프라?를 .hpp로 재구축하는게 힘들어서 그런가?
    - 사실 C++ 프로젝트면 포인터 변수랑 메모리 관리가 제일 어려울 거라 생각했는데, 그냥 다 어렵다.
        - template 사용해서 제네릭 만드는 것도 익숙치는 않고, 그래도 배워가는 거니까
        - 무슨 스마트 포인터도 써보고 있다. 사용처는 Component인데, Component에 id를 넣기를 원해서 ComponentNode를 만들고, 문제는 내부의 Component가 포인터니 Node는 delete가 안된다.<br>
        그래서 스마트 포인터를 사용하면 원래는 스택에 있는 struct ComponentNode가 사라져도 힙에 있는 Component*는 안사라지지만, 자동으로 추적해서 같이 지운다고 한다.

- ### Entity 추가
    - Entity는 플레이어나 몬스터 이런것들의 최상위 부모 느낌이다.
    - 엔티티에 컴포넌트를 붙이니까 AddComponent나 이런게 여깄다. 나중에 엔티티 말고도 써야된다면 아마 옮겨갈 수도?
    - 아직은 뭐 없다. 그냥 HP 컴포넌트만 일단 붙여놓긴 했다.

- ### Action 추가
    - C#이 정말 인프라가 잘되어있구나를 다시 느꼈다.
    - 물론 외부 라이브러리 중에 signal-slot 같은 라이브러리가 있긴 한데, delegate나 event를 직접 구현해볼 일이 얼마나 되겠냐 싶어서 그냥 직접 구현해보았다.
    - 다행히 콜백용 function은 있었다.
    - Args... 라는 것도 있었는데, 매개변수를 마치 params 처럼 여러개 받을 수 있겠다는 의미이다.
        - function<void(Args...)>는 리턴 타입은 void이고, 매개변수는 이것저것 받을 수 있는 함수야~ 라는 내용이다.
        - 그래서 신기했던게 Invoke든 Add든 구현을 하면서 매개변수에 Args...를 넣을 때 C#과 달리 타입을 강제해주지 않기 때문에, 코드에서도 딱히 오류가 안나니까 더 불안했다.
            - Action<int, string>이라고 해놨는데, Add(1.1f) 이래버려도 문제가 없는 것 아닌가 싶었기 때문이다.(실제로 빨간 줄 안그어져서..)
            - 그런데 컴파일러는 애시당초 컴파일 할 때, Args...를 알아서 타입으로 변환해서 void(int, string)로 인식한다고 한다. 신기하고 똑똑하다..
    - delegate는 += 같은 걸로 하면 되는데 이를 처음부터 구현해야 하니 AddListener랑 RemoveListener를 만들고 operator overloading을 해주었다.

내가 로직을 짜면서 이게 의도대로 실행되는지를 보여주기도 하고, SRP는 사실상 단위테스트에서 확인되기 때문에 얼마나 똑똑하게 짰는지를 볼 수 있다.<br>

제미나이의 도움을 받긴 했지만, VSCode에서 CMake로 디버거와 빌드 환경을 구성할 수 있었다.<br>

여튼 간에 테스트를 통해 발견한 버그들이 있었는데

1. 헤더의 경로 관련 문제
    - 헤더 파일들을 각각 다른 폴더에 정리해놓아 문제가 되었고, 이를 링커 매개인자로 g++을 하는 것은 많이 번거로워 CMake를 이용했다.
    - 사실 폴더의 경로부터 작성해야 한다는 사실을 애초에 처음 알았다. 컴퓨터가 똑똑하니까 루트기준 경로는 알아서 찾아주지 않을까 했지만, 인프라 구축이 필요한 일이었다.
2. 일부 네임스페이스 누락
    - std::를 안 붙인 경우가 종종 보였다.
    - cpp에서 구현 시 `using namespace std;`를 쓰는데, 헤더에는 `std::vector`같은 형식으로 일일이 적어주는 것이 원칙? 관례라고 한다.
3. StatModifier 생성 시 스택 관련 문제
    - 생성자에는 최대 스택 카운트랑 무관하게 스택 카운트를 입력하다보니 최대치를 설정해도 무시되었다.
    - 생성자 단에서도 동일하게 `min`을 통해 문제를 해결하였다.

### 이미지 렌더러
이미지 렌더링은 사실 넣을까말까 생각이 많았고 아직도 고민이다.<br>
터미널에서 작동하는 머드 게임인데 이미지를 써봐야 반쪽짜리 블록 2개를 통해 하나의 픽셀을 표현해야 하는데, 터미널 크기 특정 상 이미지 재현률이 별로이기 때문이다.<br>
그래도 뭐 언젠가 쓸지도 모르겠다 하고 일단은 이미지 렌더러 클래스를 만들어 GameManager의 인스턴스에 제공했다.

- 이미지 렌더러는 stb_image 라이브러리를 이용하여 경로 내의 이미지를 불러온다.
- 렌더링은 ftxui 라이브러리를 통해 이미지를 렌더링한다.
- 렌더링된 이미지를 다른 이벤트들과 마찬가지로 이벤트로 만들고 이벤트로 반환 구조체를 만들까 했는데, 그냥 GameContext에 같이 imageRenderer를 넣고 이용하기로 했다.
- 사실 사운드 매니저도 그렇게해도 되긴 하는데, 그냥 다양한 방식을 해보면 좋을 것 같아서 연습중이니까 해봤다.

### 기타
- 무기획으로 시작된 프로젝트라서 작업 하면서 계속 살을 덧대고 있다.
- 스탯은 DnD에서 많이 따와서 atk에서 str, dex, int로 세분화하였고, wis나 cha같은 스탯도 추가하였다.
